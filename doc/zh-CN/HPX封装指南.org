* HPX 封装指南

[[file:./readme.org][返回文档]] | [[file:../en/hpx.org][English Version]]

** 概述

=zpp/hpx= 对 HPX 做一层薄封装，让应用服务可以继续沿用 =z::server= 生命周期模型，
同时运行在 HPX runtime 之上。

推荐写法是在应用中继承 =z::hpx::server=，定义 =SVR_NAME=，最后包含
=<zpp/hpx/main.hpp>=：

#+begin_src cpp
class server : public z::hpx::server {
public:
    server(int argc, char** argv)
        : z::hpx::server(argc, argv) {
    }
};

#define SVR_NAME server
#include <zpp/hpx/main.hpp>
#+end_src

** 生命周期

=zpp/hpx/main.hpp= 会通过 =hpx::init= 启动 HPX runtime，并按以下顺序执行服务：

#+begin_src text
configure -> run -> loop -> stop -> wait_stop
#+end_src

=configure()= 或 =run()= 失败时会直接返回对应错误码，并跳过后续未启动阶段。
=run()= 成功后，即使 =loop()= 退出，也会继续执行 =stop()= 和 =wait_stop()=；清理阶段
失败时返回第一个清理错误码。

=z::hpx::run_server_lifecycle<Server>()= 负责集中编排生命周期。
=z::hpx::run_server<Server>()= 会在 server 生命周期完全结束后再 finalize HPX。

** Timer 行为

=z::hpx::server= 默认每 1000ms 调用一次 cooperative timer。它使用
=hpx::this_thread::sleep_for=，不会阻塞底层 OS worker 线程。

如果需要更快响应退出或轮询，可以在派生类构造函数中调用
=set_timer_interval(...)=。

** HPX Runtime 参数

HPX 线程数、scheduler、ini 等 runtime 参数仍通过 HPX 命令行传入。例如运行入口可以
传入：

#+begin_src sh
--hpx:threads=3
#+end_src

=doc/hpx.ini= 是面向常用 runtime 调优项的精简英文模板，
=doc/zh-CN/hpx.ini= 提供相同配置项的中文详解，包含取值范围、推荐起点、
配置依据、实际作用和参数联动关系。两份文件中的所有赋值默认均保持注释，
因此加载模板本身不会覆盖当前 HPX 构建的内建默认值。应用可以复制任一模板、
只取消所需配置项的注释，再与 zpp JSON 配置一同传入：

#+begin_src sh
./your_hpx_application server.json service \
  --hpx:config=/path/to/hpx.ini \
  --hpx:dump-config
#+end_src

需要低频、构建相关或插件专用配置时，查阅 =doc/hpx.reference.ini=。应用与 zpp
模块配置继续放在 JSON 文件中，HPX INI 只负责 HPX runtime 配置。

** 构建 HPX 示例

启用 HPX 示例：

#+begin_src sh
./zbuild.py --rebuild --with-hpx-examples
#+end_src

如果缺少 HPX package config，请先通过 zeta_forge 构建并安装 HPX。

** Sender/Receiver 适配层

=z::hpx::exec= 是针对 zeta_forge HPX 源码基线提供的 Sender/Receiver 项目级
薄适配层。应用代码应使用：

#+begin_src cpp
#include <zpp/hpx/exec/exec.hpp>

namespace ex = z::hpx::exec;

auto sender = ex::just(21) | ex::then([](int value) {
    return value * 2;
});
auto result = ex::sync_wait(std::move(sender));
#+end_src

业务代码不应直接使用 =hpx::execution::experimental=。稳定边界是
=z::hpx::exec= 显式公开的子集，而不是某个 HPX checkout 中的全部 execution
符号。该命名空间也不替代 =hpx::async=、=hpx::future=、=hpx::promise= 或
=hpx::dataflow=。

四个调度算法保留各自的底层语义：=starts_on= 选择 sender 的启动位置，
=continues_on= 转移后续 completion，=on= 保留 stdexec 的 =on= 调度契约，
=transfer= 保留独立的 HPX/stdexec 兼容契约。适配层不会用其中一个算法模拟另一个。

=from_future= 直接使用 HPX =as_sender=。=into_future= 只接受零值或单值、单一
value alternative、不含 stopped completion，并且错误类型能由 HPX =make_future=
表示的 sender；这些约束用于避免信息丢失和 HPX 在 stopped 路径上的 terminate 行为。

=sync_wait= 直接转发 HPX execution CPO，当前会使用 stdexec 的阻塞等待。不要在
唯一的 HPX worker 中等待依赖同一线程池的 sender。runtime 测试和示例至少使用两个
HPX worker，并由外部超时终止意外挂起。

启用并消费可选模块：

#+begin_src sh
./zbuild.py --with-hpx --rebuild
#+end_src

#+begin_src cmake
find_package(zpp_hpx CONFIG REQUIRED)
target_link_libraries(my_target PRIVATE zpp::hpx)
#+end_src

package 会校验当前 =zeta_forge/3rd/hpx= checkout 记录的精确 Git commit，并要求
该 HPX 使用启用 stdexec 的构建配置。
