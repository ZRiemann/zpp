* 发布历史

** v1.11.1 - Unreleased
*** 变更内容
- 新增 =nng2imp(...)=（通过 =nng2log_important(...)=），让 NNG scoped important 日志复用 =component/action= 标签并走 important 通道输出。
- 提取共享格式化逻辑 =nng2format(...)=，普通 NNG 日志与 important NNG 日志统一走 preformatted sink 路径。
- 调整 =z::log::init(...)= 中 important logger 初始化：在 async 模式下改为 =spdlog::async_logger= 并复用共享队列（blocking overflow policy），不再使用 direct logger 的逐条 flush 模式。
- 调整 =z::nng::socket= 的 pipe 生命周期日志分流：=add-post/remove-post= 进入 important 通道，=add-pre= 与 option dump 保持 debug 级别。
- 保持 important JSONL 输出 schema 与前缀结构化解析行为，同时继续保留 NNG internal logger bridge 重复消息节流策略。

*** 验证
- 扩展 async 日志测试，验证 warn/important 在共享队列中的顺序以及 JSONL important 记录中的调用线程 id 保留。
- 新增 NNG runtime 测试，验证 runtime =info= 关闭时 =nng2imp(...)= 仍能输出 important 记录。
- 新增 pipe 生命周期测试，验证 =add-post/remove-post= 在 runtime 非 debug 级别下仍进入 important 通道，而 =add-pre= 不进入 important 输出。

** v1.11.0 - 2026-07-13
*** 变更内容
- 扩展 =z::log::config=，在保留旧 rotating runtime file 字段兼容的同时，新增结构化 important file 日志配置。
- 新增独立 important 通道入口 =spd_imp(...)=，将普通 =info= 记录同时写入 runtime 日志和 important JSONL 日志，不额外增加 =important= JSON 字段。
- 新增 =nng2imp(...)=，使 NNG scoped important 记录复用现有 component/action tag 格式。
- async 模式下 important 记录复用 spdlog 队列，保持与 runtime 记录的队列提交顺序，并避免在 NNG 回调内逐条同步 flush。
- 已建立和已移除的 NNG pipe 作为 important 生命周期事件记录，不再受 runtime debug 级别影响；add-pre 明细和 option dump 继续保持 debug 级别。
- important 日志支持 JSONL formatter，稳定输出 service、node、run、进程/线程、源码位置、component、action 和 message 字段。
- 将 =[ctp-md:login-rsp]=、=[nng:socket:create]= 等常见消息前缀解析为结构化 =component= 和 =action=。
- =z::server= 的 =spd= JSON 解析新增 =service=、=node=、=run_id= 和 =important= sink 设置。
- 在 zpp logger bridge 中按 level、facility、message id 和 message 对重复的 NNG 内部日志做节流，使 important 日志保留首条记录和周期性摘要，而不是每次重试都写入。

*** 验证
- 新增日志测试覆盖 =warn=+ JSONL important sink、=spd_imp(...)= 直写 info 记录、旧 runtime file 兼容、async logger 下保留 zpp 短线程 id，以及共享队列中的 warn/important 顺序。
- 新增 NNG runtime 日志测试，覆盖重复内部日志会被抑制，不同 level 或不同 message 仍正常透传，验证 =nng2imp(...)= 在 runtime =info= 关闭时仍写入 important 通道，并验证 pipe 生命周期事件的 important/debug 分流。

** v0.9.0 - 2026-07-10

*** 验证

** v0.7.0 - 2026-07-05
*** 发布定位摘要
- 从 =v0.6.0= 升级到 =v0.7.0=，核心是可复用 NNG 协议封装与仓库级 C++ 格式规范。
- 对 =zpp::nng= 进行破坏性 API 清理，移除过时兼容接口并统一协议生命周期与所有权语义。

*** 变更内容
- 新增仓库级 =.clang-format= 配置与 VS Code formatter 设置，并按统一格式整理 zpp C++ 源码、头文件、示例和测试。
- 将所有公开协议封装统一到 =zpp/nng/protocols/=，新增聚合头 =protocols.h=；直接移除旧 =zpp/nng/publisher.h= 与 =zpp/nng/subscriber.h= 路径，不提供转发兼容头。
- 新增公共 =endpoint_protocol= 生命周期复用，并完成 PUSH/PULL、REQ/REP、SURVEYOR/RESPONDENT、BUS 及单 endpoint PAIR0/PAIR1 封装；REQ/REP 与 SUB 收敛为管理式异步并发接口，其余协议按合法方向提供同步与 AIO 接口。
- 将 =aio_ctx= 增强为可移动、事务式 RAII，仅保留异步 =send(message&)= 与 =recv()= 操作；由 REQ/REP 内部管理并发 context，SURVEYOR、RESPONDENT 继续提供应用管理的 context 和 survey timeout 等协议专用配置。
- 新增 =z::nng::context= 薄封装，覆盖 =nng_ctx= 生命周期、同步/AIO 收发、当前 NNG 2.0 typed context option，以及公共收发超时的批量配置，并由 =aio_ctx= 组合复用。
- 新增支持事务式多 endpoint 配置的 =publisher= 与 =subscriber=，option 组均可省略，单个 socket 可混合 listen/dial role，生命周期顺序由调用方保证；subscriber 在默认 context 上管理固定 receive AIO 池，确保每条匹配消息只投递一次。
- 将宽泛的 =event_forwarder= API 替换为 =zpp/nng/protocols/= 下的 =pub_sub_device=，device completion 改用 RAII =aio= 封装，在 NNG 接管后释放 socket wrapper 所有权，并移除冗余的 =running()= 状态查询。
- 将 listener/dialer endpoint 创建和启动逻辑收敛为 =endpoint_protocol= 私有实现，由全部协议封装内部复用。
- 将 endpoint 元数据校验归入 =zpp/nng/endpoint.h=，并完成 socket、listener、dialer 的批量 =set_options/get_options=；endpoint 与 TCP transport option 组可独立省略。
- 将同步 PUB send 热路径改为内联薄转发，保留 NNG 错误码和明确的消息所有权语义。
- 将 PUB/SUB 程序重命名为 =publisher_example= 与 =subscriber_example=，新增 =pub_sub_device_example=，演示 publisher 经 device 到 subscriber 的三进程透传拓扑。
- 新增可运行的 PUSH/PULL、REQ/REP、PAIR1 listen/dial 示例，共享 JSON option 解析与可复用 AIO 状态机，并补充对应 =doc/server.json= 配置。
- 将 REQ/REP 生命周期统一为 =start(context_count)= 与 =stop()=：启动时自动创建并发 context，停止时先关闭 AIO/context 再关闭 endpoint；移除公开的 =open_context()/send()/recv()= 及 =start_mt()/stop_mt()=。requester 通过 =request()/on_reply()= 和 =hint= 关联乱序应答。
- 将 SUB 生命周期统一为 =start(aio_count)= 与 =stop()=：在 socket 单一默认 context 上提交多个 receive，callback 完成后自动重投对应 AIO，停止时先关闭全部 AIO 再关闭 endpoint；移除公开的 =subscribe()/unsubscribe()/open_context()/recv()=，通过可并发执行的 =on_receive(message&, result)= 扩展点交付消息。
- 清理 =pipe= option getter 的指针输出重载，并移除 NNG 2.0 已不再提供实现的 dialer/listener option wrapper。
- 简化 =nng::message= 所有权状态：=msg_= 非空时始终由封装层拥有，发送成功或 =release()= 后置空，并移除独立的 =owned_= 标记。
- 为协议框架补充按阶段定位的错误日志，覆盖配置、endpoint 启动、托管 AIO/context 创建、callback 状态异常，以及 requester context 池初始化和回收失败；正常背压与有序停止仍不记录为错误。

*** 兼容性与升级说明
- PUB/SUB include 需迁移到 =zpp/nng/protocols/publisher.h= 与 =zpp/nng/protocols/subscriber.h=，也可使用聚合头 =zpp/nng/protocols/protocols.h=。
- REQ/REP 调用方应按 =configure() -> start(context_count) -> request/serve -> stop()= 管理生命周期，不再单独调用 =start_mt()/stop_mt()=；SURVEYOR 和 RESPONDENT 的应用管理 context 在 =stop()= 前仍须先关闭。
- SUB 调用方应按 =configure() -> start(aio_count) -> receive callback -> stop()= 管理生命周期。原有直接 =subscribe()/unsubscribe()/open_context()/recv()= 调用应迁移到配置 topics 和 =on_receive()=；确需底层操作时直接使用 =z::nng::socket=、=context= 与 =aio_ctx=。
- 原有 REQ/REP =open_context()/send()/recv()= 调用应迁移到 requester 的 =request()/on_reply()= 或 replier 的 =on_receive()/on_send()= 扩展点；context 级底层操作可直接使用 =z::nng::context= 与 =aio_ctx=。
- 原有同步或裸消息 =aio_ctx= 收发调用应改用 =context= 完成同步 I/O，或改用 =aio_ctx::send(message&)= 与 =recv()= 完成异步 I/O。异步发送在提交时转移所有权；发送失败后通过 =io().release_msg()= 取回消息。重复转发的 =aio_ctx::get()/id()= 已移除，context handle 和 id 统一通过 =ctx().get()= 与 =ctx().id()= 访问。
- 将 =event_forwarder= 及其头文件迁移为 =zpp/nng/protocols/pub_sub_device.h= 中的 =pub_sub_device=；旧类型、include 路径和 =running()= 不提供兼容别名。
- PUB/SUB 示例配置必须提供一个 =ingress_endpoint=、非空 =egress_endpoints= 数组和非空 =topics= 数组；空 topic 字符串表示订阅全部消息。
- 原有 =socket::apply_options()= 调用需改为 =socket::set_options()= 并处理返回的 NNG 错误码。
- 原先调用已移除 option wrapper 的代码需要改用当前 NNG 2.0 支持的接口。

** v0.5.2 - 2026-05-18
*** 发布定位摘要
- zpp 不再维护或运行自己的 Conan 配方，改为消费 =zeta_forge deps= 安装的共享依赖环境。
- 明确职责边界：=zeta_forge= 作为唯一第三方依赖安装与配置管理入口，zpp 只消费该环境并输出自身库。
- 关闭 zpp 构建链路中的 fallback/legacy 分支，改为失败即明确提示先完成 zeta_forge 环境配置。
- 按补丁版本发布，zpp 从 =v0.5.1= 升级到 =v0.5.2=。

*** 变更内容
- 删除 =builder/conanfile.py=，=builder/zpp.py= 配置阶段只把 =${ZETA_INSTALL_PREFIX}/lib/cmake/zeta_deps/<BUILD_TYPE>= 加入 =CMAKE_PREFIX_PATH=。
- 顶层 CMake 对 =RapidJSON=、=spdlog=、=GTest= 使用 =find_package(... CONFIG REQUIRED)=，找不到时提示先安装 =zeta_forge deps=。
- =builder/zpp.py= 不再校验或注入 =ZETA_RAPIDJSON_SRC_DIR=/RAPIDJSON_ROOT=，RapidJSON 来源统一由 =zeta_forge deps= 安装的 package config 提供。
- =builder/zpp.py= 配置阶段清理旧 build cache 中的 =CMAKE_TOOLCHAIN_FILE= 和遗留 =conan_toolchain.cmake= CMake system state，避免历史 Conan toolchain 污染当前共享依赖模式。
- =zppConfig.cmake= 在恢复 PUBLIC 依赖 target 前自动加入安装前缀中的 =zeta_deps/<BUILD_TYPE>=，并通过 =find_dependency(RapidJSON CONFIG)= 恢复 =rapidjson::rapidjson=。
- 顶层 =CMakeLists.txt= 移除 =ZETA_CMAKE_UTIL_DIR= 的 =$ZETAX_ROOT= fallback 自动回填逻辑。
- 顶层 =CMakeLists.txt= 移除 =ZETA_DEPS_CMAKE_DIR= 基于 =CMAKE_INSTALL_PREFIX= 的 fallback 自动推导逻辑。
- =builder/zpp.py= 移除历史 Conan build state 探测与自动清理分支，不再隐式迁移旧构建目录。
- =builder/zpp.py= 配置命令移除遗留 Conan 兼容参数 =-U CMAKE_TOOLCHAIN_FILE=。
- 导出 =zppConfig.cmake= 时，若安装前缀下缺少 =zeta_deps/<BUILD_TYPE>= 将直接失败并提示先执行 =zeta_forge/zbuild.py deps --install=。
- =readme.org= 与 =doc/zh-CN/readme.org= 同步更新为"先准备 zeta_forge 环境，再构建 zpp"的统一契约说明。
- zpp CMake project version 与仓库根 =VERSION= 升级到 =v0.5.2=。
- 顶层 CMake 改为从仓库根 =VERSION= 读取并校验 =vX.Y.Z=，避免 =project(zpp VERSION ...)= 与发布版本漂移。

*** 兼容性与升级说明
- 构建 zpp 前需要先执行 =$ZETAX_ROOT/zeta_forge/zbuild.py deps --BUILD_TYPE=<type> --install=。
- zpp 不再从自身 build tree 暴露 Conan generators，下游项目也不应引用 =zpp/build/.../conan=。
- 推荐且默认支持的入口是 =./zbuild.py=。
- 若使用 direct CMake，需显式传入 =-DZETA_CMAKE_UTIL_DIR= 与 =-DZETA_DEPS_CMAKE_DIR=。
- 从旧 Conan 构建目录升级时请先清理旧 build tree（如 =rm -rf build/<BUILD_TYPE>=），避免历史缓存干扰。
- 版本升级只需要先更新 =VERSION=，CMake project version 会在 configure 阶段自动同步。
