* Release History

** v0.6.0 - 2026-06-08
*** Release Positioning Summary
- Feature release from =v0.5.3= to =v0.6.0=, expanding zpp's reusable NNG runtime and system utility surface.
- Continue extracting downstream-agnostic NNG building blocks into the optional =zpp_nng= module.
- Replace the legacy wall-clock helper with explicit monotonic timer and wall-time formatting utilities.

*** Changes
- Add =zpp/nng/endpoint.h= and =src/nng/endpoint.cpp= for common NNG URL transport parsing, endpoint roles, TCP URL formatting, token replacement, IPC path extraction, and stale IPC socket cleanup.
- Refactor =zpp/nng/socket.h= into a movable RAII socket owner with protocol open helpers, socket option accessors, listen/dial wrappers, raw PUB/SUB support, and compatibility aliases for historical =pare*= methods.
- Add =zpp/nng/runtime.h= and =src/nng/runtime.cpp= for runtime socket options, nanosecond timestamps, normal PUB publishing, and raw SUB-to-PUB event forwarding.
- Add =zpp_nngConfig.cmake= export support so downstream projects can restore =zpp= and =nng= dependencies from installed package configs.
- Add =zpp/system/timer.hpp= and =zpp/system/wall_time.hpp=, and remove the legacy =zpp/system/time.h= helper.
- Extend =z::sys= with filesystem availability, machine UUID lookup, POSIX directory creation, rename, and callback-driven directory tree walking/deletion utilities.
- Add =./zbuild.py --all=, always generate =compile_commands.json=, and make Taskflow examples prefer =Taskflow::Taskflow= config packages with =TASKFLOW_ROOT= fallback.
- Normalize queue, engine, coroutine, NNG, Folly, and Taskflow internals to trailing-underscore member names, and fix scheduler thread-local dispatch cursor initialization.

*** Verification
- Add =gtest_nng= coverage for endpoint helpers and the new runtime socket/publisher wrappers.
- Add timer, wall-time, legacy wall-time, system, and STL queue coverage for the new utility APIs and renamed internals.

** v0.5.3 - 2026-05-28
*** Release Positioning Summary
- Stabilize the =zpp/hpx= thin lifecycle wrapper while keeping =z::zhpx::server= a lightweight subclass of =z::server=.
- Fix HPX entrypoint error-code propagation and server destruction timing, leaving clear boundaries for later HPX extensions.

*** Changes
- Add =z::zhpx::run_server_lifecycle<Server>()= to centralize the =configure/run/loop/stop/wait_stop= lifecycle and return the first failure error code.
- Add =z::zhpx::run_server<Server>()= in =zpp/hpx/main.hpp= so =hpx::finalize()= is called only after the server lifecycle has fully ended.
- After =run()= succeeds, =stop()= and =wait_stop()= always run regardless of how =loop()= exits; when =configure()= or =run()= fails, the original error code is preserved and later unstarted phases are skipped.
- Add configurable timer interval support to =z::zhpx::server=, keeping the default at 1000ms and continuing to use =hpx::this_thread::sleep_for= for cooperative sleep.
- Narrow HPX includes in =zpp/hpx/server.hpp= and add direct =std::mutex= and =spdlog::details::null_mutex= includes to =zpp/hpx/spdlog.hpp=.
- Add HPX wrapper entrypoint, error-code semantics, and timer behavior documentation.
- Make =./zbuild.py run= wait for run targets to exit gracefully on Ctrl-C, avoiding a Python parent-process traceback before HPX/zpp services can handle the signal.

*** Verification
- Add =hpx_server_lifecycle_test.cpp= to cover =configure/run/stop/wait_stop= failure paths and the success path.

** v0.5.2 - 2026-05-18
*** Release Positioning Summary
- zpp no longer maintains or runs its own Conan recipe, and now consumes the shared dependency environment installed by =zeta_forge deps=.
- Clarify ownership boundaries: =zeta_forge= is the only third-party dependency installation and configuration entrypoint; zpp only consumes that environment and exports its own library.
- Close fallback and legacy branches in the zpp build path, failing with clear guidance to prepare the zeta_forge environment first.
- Patch release from =v0.5.1= to =v0.5.2=.

*** Changes
- Remove =builder/conanfile.py=; =builder/zpp.py= now only prepends =${ZETA_INSTALL_PREFIX}/lib/cmake/zeta_deps/<BUILD_TYPE>= to =CMAKE_PREFIX_PATH= during configure.
- Top-level CMake now uses =find_package(... CONFIG REQUIRED)= for =RapidJSON=, =spdlog=, and =GTest=, and reports that =zeta_forge deps= must be installed first when missing.
- =builder/zpp.py= no longer validates or injects =ZETA_RAPIDJSON_SRC_DIR=/RAPIDJSON_ROOT=; RapidJSON comes only from the package config installed by =zeta_forge deps=.
- =builder/zpp.py= clears stale =CMAKE_TOOLCHAIN_FILE= and legacy =conan_toolchain.cmake= CMake system state from old build caches to avoid historical Conan toolchain pollution.
- =zppConfig.cmake= automatically adds the install-prefix =zeta_deps/<BUILD_TYPE>= path before restoring PUBLIC dependency targets, and restores =rapidjson::rapidjson= through =find_dependency(RapidJSON CONFIG)=.
- Remove the =$ZETAX_ROOT= fallback auto-fill for =ZETA_CMAKE_UTIL_DIR= from the top-level =CMakeLists.txt=.
- Remove the =CMAKE_INSTALL_PREFIX=-based fallback inference for =ZETA_DEPS_CMAKE_DIR= from the top-level =CMakeLists.txt=.
- Remove legacy Conan build-state detection and automatic cleanup branches from =builder/zpp.py=; old build directories are no longer implicitly migrated.
- Remove the legacy Conan compatibility =-U CMAKE_TOOLCHAIN_FILE= configure argument from =builder/zpp.py=.
- Exported =zppConfig.cmake= now fails directly if the install prefix lacks =zeta_deps/<BUILD_TYPE>=, with guidance to run =zeta_forge/zbuild.py deps --install= first.
- Update README documentation to describe the unified contract: prepare zeta_forge first, then build zpp.
- Upgrade zpp CMake project version and root =VERSION= to =v0.5.2=.
- Top-level CMake now reads and validates =vX.Y.Z= from the root =VERSION= file to prevent drift from =project(zpp VERSION ...)=.

*** Compatibility and Upgrade Notes
- Build zpp only after running =$ZETAX_ROOT/zeta_forge/zbuild.py deps --BUILD_TYPE=<type> --install=.
- zpp no longer exposes Conan generators from its own build tree; downstream projects should not reference =zpp/build/.../conan=.
- The recommended and default supported entrypoint is =./zbuild.py=.
- Direct CMake usage must explicitly pass =-DZETA_CMAKE_UTIL_DIR= and =-DZETA_DEPS_CMAKE_DIR=.
- When upgrading from an old Conan build directory, clean the old build tree first, for example =rm -rf build/<BUILD_TYPE>=.
- Version upgrades only need to update =VERSION=; CMake project version syncs automatically at configure time.

** v0.5.1 - 2026-05-13
*** Release Positioning Summary
- Fix zpp logging short thread IDs, queue chunk lifecycle, and Taskflow pipeline runtime stability.
- Patch release from =v0.5.0= to =v0.5.1=.

*** Changes
- Change the =uni_svr= run entry to millisecond-level waiting and avoid misleading error-level example logs; the entry can exit on its own when complete.
- zpp logging macro pattern =%t= now reuses the two-digit short thread ID from =z::tid::id()=, avoiding overly long system thread IDs and preserving column alignment; direct native =g_spd_log->log(...)= calls still keep spdlog's default thread-id semantics.
- Clear =next= links when initializing and reusing =spsc/mpmc= chunks, avoiding heap corruption from reading uninitialized pointers during queue destruction before expansion.
- Restore =pipeline_mpsc= to reuse =z::mpmc=, and wait in =wait_done()= until the Taskflow run callback fully exits to avoid early destruction of local pipelines.
- Upgrade zpp CMake project version and root =VERSION= to =v0.5.1=.

** v0.5.0 - 2026-05-13
*** Release Positioning Summary
- Add project run-entry CLI support to list and execute preset =add_run_target= commands.
- Feature release from =v0.4.1= to =v0.5.0=.

*** Changes
- Add =runs= and =run <name>= subcommands to =zbuild.py=, reusing =zeta_forge.run_targets= to scan and execute in-project =add_run_target(...)= entries.
- =runs= prints run-entry names and equivalent CMake commands in a concise readable list.
- =run <name>= directly executes already built binaries with their preset arguments from the existing build tree; it does not configure the project or call =cmake --build=.
- Upgrade zpp CMake project version and root =VERSION= to =v0.5.0=.

** v0.4.1 - 2026-05-13
*** Release Positioning Summary
- Optimize zpp logging macro cost for disabled levels while preserving historical entries such as =spd_inf=.
- Patch release from =v0.4.0= to =v0.4.1=.

*** Changes
- =spd_inf/spd_err/spd_war/spd_dbg/spd_trc= now obey =SPDLOG_ACTIVE_LEVEL=, allowing explicitly disabled log levels to be compiled out.
- Logging macros add runtime short-circuit checks so disabled levels do not evaluate log arguments when backtrace is not enabled.
- =spd2*= printf logging macros reuse the lightweight runtime short-circuit path to avoid argument evaluation and =vsnprintf= formatting for disabled levels.
- =z::server= logging configuration now directly overrides =z::log::config= defaults, reducing temporary configuration objects while preserving invalid-value guards.
- Keep the thin =spd_logx= template wrapper and the native spdlog hot path through =g_spd_log->log(...)=.

*** Verification
- Add logging unit tests for runtime short-circuiting, backtrace cache preservation, printf short-circuiting, and =SPDLOG_ACTIVE_LEVEL= compile-time trimming.

** v0.3.1 - 2026-05-11
*** Release Positioning Summary
- Complete migration from the old =zpp/json.h= to the mature =zpp/json.hpp= API and hard-delete the old header.
- Patch release from =v0.3.0= to =v0.3.1=; the fortune project version remains unchanged.

*** Changes
- zpp core, nng engine, and thread_pool test services now use the =json.hpp= =member=, =get=, =size=, and =at= APIs for configuration reads.
- fortune quotations configuration reads now use =json.hpp=, and the message-queue configuration entry type changes to =z::json_doc*=.
- Delete the old =include/zpp/json.h= header, and remove the old =tests/uni_svr/test_json.cpp= and =test_json()= entry.
- zpp package config restores PUBLIC dependency targets such as =Threads=, =rapidjson::rapidjson=, and =spdlog::spdlog= before importing targets, reducing explicit dependency-order requirements for downstream projects.
- =zbuild.py --with-hpx-examples= now injects the =AsioConfig.cmake= compatibility path from zeta_forge HPX to avoid =find_package(HPX)= failures caused by missing Asio.
- Folly/NNG examples now link =zpp::folly= / =zpp::zpp_nng= module targets, letting zpp modules propagate third-party dependencies and avoiding sibling directories missing local imported targets.

*** Compatibility and Upgrade Notes
- The old =json.h= compatibility shim is not kept; applications must include =<zpp/json.hpp>= and use the new API.
- Prefer rvalue =set/push= when constructing or inserting temporary JSON child objects; use copy semantics only when the source object must be reused.
- =rapidjson= remains the underlying dependency of =json.hpp= and is not removed from zpp core dependencies.

*** Verification

- =json_hpp_test.cpp= continues to cover parsing, construction, move insertion, traversal, deletion, and non-object JSON file-loading scenarios.

** v0.3.0 - 2026-05-11
*** Release Positioning Summary
- Add deletion APIs to =z::json=, expanding the public API from configuration read/build support to object-member and array-item deletion.
- Per semantic versioning, this interface addition upgrades from =v0.2.x= to =v0.3.0=.
- Fix zpp Debug builds incorrectly mapping Conan imported targets to Release configuration.

*** Changes
- =z::json::remove(const std::string &key)= deletes the member for the given object key.
- =z::json::remove(size_t index)= deletes the item at the given array index and shifts following elements forward.
- Deletion error-code contract:
  - Return =ERR_NOT_SUPPORT= when the current JSON value is not the corresponding object/array type.
  - Return =ERR_NOT_EXIST= when an object key does not exist.
  - Return =ERR_OUT_OF_RANGE= when an array index is out of range.
- Add =json_hpp_test.cpp= scenarios for object-member deletion, array-item deletion, missing keys, out-of-range indices, and unsupported types.
- Remove the incorrect global =CMAKE_MAP_IMPORTED_CONFIG_DEBUG=Release= mapping from =builder/zpp.py= and clear old =CMAKE_MAP_IMPORTED_CONFIG_DEBUG= cache values during configure.
- Add =zbuild.py= and =builder/zpp.py= to configure dependencies in =builder/zpp.py= so build-script changes trigger reconfigure.

*** Compatibility and Upgrade Notes
- Existing =set=, =get=, =member=, =at=, =push=, and iteration APIs remain compatible.
- New deletion APIs do not change missing-value read behavior; after deletion, =get/member/at= continue to return existing error codes.
- Debug builds now consume Debug Conan dependency configs and no longer depend on Release imported-target mapping.

*** Verification
- =./zbuild.py --BUILD_TYPE Debug= passed.
- =./build/Debug/tests/unit_tests/gtest_tsc= passed; all 13 tests passed.

** v0.2.1 - 2026-05-11
*** Release Positioning Summary
- Adapt zpp into a lightweight default build for open-source users: only core builds by default.
- Heavy dependencies such as =folly=, =nng=, and =taskflow= become optional modules and are disabled by default.
- Dependency lookup moves down into the CMake subdirectories that actually compile each module; the top-level =CMakeLists.txt= no longer forces those packages to be found.

*** Changes
- Add CMake options =ZPP_BUILD_FOLLY_MODULE=, =ZPP_BUILD_NNG_MODULE=, and =ZPP_BUILD_TASKFLOW_MODULE=, all defaulting to OFF.
- Top-level =CMakeLists.txt= keeps only core required dependencies: =Threads=, =rapidjson=, and =spdlog=.
- Remove top-level forced =find_package(folly)=, =find_package(nng)=, and =find_package(taskflow)= calls.
- Move dependency lookup:
  - =folly= lookup moves to =src/folly/CMakeLists.txt= and runs only when =ZPP_BUILD_FOLLY_MODULE=ON=.
  - =nng= lookup moves to =src/nng/CMakeLists.txt= and runs only when =ZPP_BUILD_NNG_MODULE=ON=.
  - =taskflow= lookup moves to =examples/taskflow/CMakeLists.txt= and =examples/hpx/taskflow/CMakeLists.txt=, and runs only when =ZPP_BUILD_TASKFLOW_MODULE=ON=.
- =src/CMakeLists.txt= now adds Folly and NNG modules conditionally.
- =examples/CMakeLists.txt= now adds Folly, NNG, and Taskflow examples conditionally.
- =examples/hpx/CMakeLists.txt= now builds HPX Taskflow examples only when both =ZPP_BUILD_HPX_EXAMPLES=ON= and =ZPP_BUILD_TASKFLOW_MODULE=ON=.
- =tests/CMakeLists.txt= now builds coroutine tests, which depend on Folly, only when =ZPP_BUILD_FOLLY_MODULE=ON=.
- =tests/unit_tests/CMakeLists.txt= now builds Folly unit tests only when =ZPP_BUILD_FOLLY_MODULE=ON=.
- Add =--with-folly=, =--with-nng=, and =--with-taskflow= CLI options to =builder/zpp.py=, all disabled by default.
- =builder/zpp.py= now conditionally injects =CMAKE_PREFIX_PATH=, =folly_DIR=, =OpenSSL_DIR=, and =TASKFLOW_ROOT= only when the corresponding module is enabled.

*** Compatibility and Upgrade Notes
- Core default build path: =./zbuild.py --rebuild= builds only the =zpp::zpp= core library without extra dependencies such as Folly, NNG, or Taskflow.
- Enable optional modules with =./zbuild.py --rebuild --with-folly --with-nng --with-taskflow=.
- =ZPP_BUILD_EXAMPLES=ON= does not automatically enable optional modules; examples for disabled modules are skipped automatically.
- Existing variables such as =ZETA_ZPP_SRC_DIR= and =ZETA_TASKFLOW_SRC_DIR= remain compatible.

*** Verification
- Core default build: =./zbuild.py --BUILD_TYPE=Debug --no-tests --no-examples --rebuild= passed without running =find_package(folly)=, =find_package(nng)=, or =find_package(taskflow)=.
- Folly-only module: =./zbuild.py --BUILD_TYPE=Debug --with-folly --no-tests --no-examples --rebuild= passed.
- NNG-only module: =./zbuild.py --BUILD_TYPE=Debug --with-nng --no-tests --no-examples --rebuild= passed.
- Taskflow-only module: =./zbuild.py --BUILD_TYPE=Debug --with-taskflow --no-tests --no-examples --rebuild= passed.
- Combined modules: =./zbuild.py --BUILD_TYPE=Debug --with-folly --with-nng --with-taskflow --no-tests --no-examples --rebuild= passed.

** v0.2.0 - 2026-05-11
*** Release Positioning Summary
- Migrate zpp from the legacy bundled CMake helper/submodule build mode into an independent ZetaX project.
- zpp now maintains project-level =builder/zpp.py= and =builder/conanfile.py= while reusing the shared =zeta_forge= build lifecycle.
- =cmake_util= stops being a zpp dependency submodule; CMake helpers now come from =$ZETAX_ROOT/zeta_forge/cmake_util=.

*** Changes
- Add lightweight =zbuild.py=, locating the zeta_forge shared Python library through =$ZETAX_ROOT/zeta_forge= or sibling workspace paths.
- Add =builder/zpp.py=, inheriting =zeta_forge.cmake_builder.CMakeProjectBuilder= and injecting the zpp source directory, Conan recipe, CMake options, and test/example switches.
- Add =builder/conanfile.py= for zpp project-private Conan dependencies.
- Remove the =cmake_util= submodule and old =.gitmodules= binding.
- Change =CMakeLists.txt= to include public CMake helpers through =ZETA_CMAKE_UTIL_DIR= and prefer dependency environments injected by zeta_forge.
- Debug zpp builds can consume Release Folly dependencies deployed by forge through =CMAKE_MAP_IMPORTED_CONFIG_DEBUG=Release= imported-target mapping.
- Unify zpp CMake project version and package version at =0.2.0=, and add root =VERSION=.

*** Compatibility and Upgrade Notes
- Recommended command: =cd $ZETAX_ROOT/zpp && ./zbuild.py --rebuild=.
- The old =cmake_util= submodule and =cbuild= path are no longer maintained as recommended build methods.
- Ensure =$ZETAX_ROOT/zeta_forge= exists and baseline dependencies are prepared through the zeta_forge toolchain before building.
- Source path override variables such as =ZETA_ZPP_SRC_DIR=, =ZETA_TASKFLOW_SRC_DIR=, and =ZETA_RAPIDJSON_SRC_DIR= remain supported.

*** Verification
- =python3 -m py_compile zbuild.py builder/zpp.py= passed.
- =./zbuild.py --help= passed.
- =./zbuild.py --BUILD_TYPE=Debug --no-tests --no-examples --rebuild= passed, completing Conan install, CMake configure, and static-library build.
